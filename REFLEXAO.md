# O que foi difícil 

Uma das partes mais difíceis foi entender como funcionam as branches na prática, principalmente a relação entre a branch master e as outras branches criadas. No início, fiquei confusa sobre quando as mudanças realmente eram aplicadas na master, já que o Pull Request não faz o merge automaticamente. 

Outro ponto complicado foi lidar com o conflito de merge. No começo, ver aqueles símbolos foi confuso, e eu não entendia exatamente o que precisava ser feito. Precisei prestar bastante atenção para não apagar coisas importantes e entender qual versão deveria manter. 

Além disso, também tive dificuldades com alguns comandos do Git no PowerShell, como criar arquivos e lidar com pequenos erros de sintaxe. 

# O que ficou claro 

Ficou muito claro para mim como funciona o fluxo básico do Git: criar uma branch, fazer alterações, commitar, enviar para o GitHub e depois abrir um Pull Request. Entendi que o Pull Request é uma forma de revisar mudanças antes de integrar ao código principal. 

Também entendi bem a importância de usar commits semânticos, pois eles ajudam a organizar melhor o histórico do projeto e deixam claro o que foi feito em cada etapa. 

Outro ponto que ficou claro foi que o merge precisa ser feito manualmente após o Pull Request, e que a branch master só é atualizada depois disso. 

# O que ainda é confuso 

Ainda tenho algumas dúvidas sobre situações mais complexas envolvendo Git, como quando existem muitos conflitos ao mesmo tempo ou quando várias pessoas estão trabalhando no mesmo projeto. 

Também não entendi completamente todos os detalhes internos de como o Git gerencia as versões e o histórico, como o funcionamento exato do HEAD em diferentes situações. 

Apesar disso, consegui realizar todas as etapas do exercício seguindo o processo corretamente, mas sei que ainda preciso praticar mais para ter mais segurança.
